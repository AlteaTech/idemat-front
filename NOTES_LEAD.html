Questions & arbitrages en attente

  • 1
    5 écrans sans ticket Kanban Les écrans "Recharger mon compte", "Hub Mon compte", "Paramètres du compte", "Modifier email" et "Modifier mot de passe" ont été développés sans issue dans le Kanban. Faut-il créer des tickets rétroactivement (IV-XX) ?
  • 2
    PUT /profil — nom et prénom combinés ou séparés ? La maquette montre un champ unique "Nom, Prénom". Le modèle front envoie actuellement nomPrenom: string. L'API doit-elle recevoir un champ unique ou nom + prenom séparés ?
  • 3
    Texte des conditions générales (inscription) Dans le .NET, le texte venait de strMentionsLegales (base de données, par enseigne). Viendra-t-il d'un endpoint API ou sera-t-il statique côté front ?
  • 4
    Redirection post-inscription Après envoi du formulaire d'inscription, on redirige vers /demande-ok (page de confirmation). Est-ce le comportement souhaité, ou doit-on rediriger vers /connexion-idemat ?
  • 5
    "J'ai oublié mon mot de passe" — passer le contrat ? Le lien depuis la connexion navigue vers /mot-de-passe-oublie sans le paramètre :contrat. Faut-il le passer (/mot-de-passe-oublie/test) pour charger le bon logo enseigne ?
  • 6
    Ville (dropdown ou libre) pour les professionnels Pour les particuliers, la ville est un dropdown depuis les communes du contrat. Pour les professionnels, une entreprise peut être hors du périmètre. Doit-on laisser un champ texte libre pour les pros ?
  • 7
    Options d'achat de passages — viennent du BO ? L'écran "Recharger mon compte" affiche des options (nombre de passages + tarif). Ces options sont-elles configurées dans le BO par enseigne, ou sont-elles fixes côté front ?
  • 8
    Suppression de compte Le bouton "Supprimer le compte" existe dans l'écran Paramètres mais la confirmation et l'appel API ne sont pas implémentés (TODO). À préciser : flux de confirmation, délai de grâce, etc.

Choix techniques notables

  • Services agents mockés — tous les appels API IDemat sont dans src/services/agents/idemat/ avec of(mockData).pipe(delay(...)). Le remplacement par de vrais appels HTTP sera localisé dans ces fichiers uniquement.
  • Client OpenAPI idbatv7 conservé — src/core/api/ contient le client auto-généré pour le futur BO IDemat. Ne pas supprimer.
  • Hauteur mat-form-field — Angular Material fixe les champs à 56px par défaut. Pour les réduire à 44px sans casser les labels flottants, le pattern ::ng-deep .mdc-text-field--outlined { height: 44px } + repositionnement mdc-floating-label est nécessaire (voir modification-profil.component.scss).
  • Inscription — composant unique — un seul InscriptionComponent gère particuliers et professionnels via le paramètre de route /:type. Les sections conditionnelles (Civilité, Carte d'identité, Justificatif pour les particuliers ; Société + SIRET pour les pros) sont contrôlées par type().
  • Autofill navigateur — le bouton "Se connecter" n'est pas désactivé sur form.invalid pour éviter le problème d'autofill (Angular ne détecte pas les valeurs remplies automatiquement). La validation est faite côté onLogin().
  • Nettoyage skeleton — 49 fichiers legacy idbatv7 BO supprimés (services agents, modèles, composants shared dialogs). Les intercepteurs error.interceptor et loader.service ont été simplifiés.
