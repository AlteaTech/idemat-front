Contrainte BDD — à lire avant tout

  • Pas de ALTER TABLE ni CREATE TABLE
  • Pas de fichier de migration (EF Core, Flyway, Liquibase…)
  • Pas de renommage ou suppression de colonne
  • Si une donnée semble manquer en base : remonter au lead avant d'improviser une colonne
La base IDemat est en production côté .NET. Toute modification de structure casse l'application existante. Le dev back cartographie les tables existantes et s'y tient.

Ordre d'implémentation suggéré

  • Auth — POST /auth/login Nécessaire pour tout le reste. Retourne le JWT utilisé par tous les endpoints sécurisés.
  • Contrat — GET /contrat/by-url/{url} Chargé dès la mire de connexion pour afficher le logo de l'enseigne. Aucune auth requise.
  • Inscription — POST /inscription Priorité lead Route publique, multipart. Priorité explicite dans le message du lead. Inclure la validation côté serveur (email unique, SIRET format, etc.).
  • Profil usager — GET /usager/me + PUT /usager/profil + véhicules Débloque tous les écrans "Mon compte". Cadrer d'abord la question nomPrenom combiné vs séparé.
  • Passages & déchetteries Endpoints en lecture seule, peu de complexité métier.
  • Achat de passages (PayFip) Dépend de la configuration PayFip et des décisions métier sur les options d'achat. À cadrer avec le lead avant d'implémenter.
  • Reset mot de passe — POST /usager/reset-password Nécessite la configuration d'un SMTP / service d'envoi d'e-mail. À traiter en parallèle de l'inscription.
Côté front : tous les appels sont dans src/services/agents/idemat/, chacun avec un commentaire // TODO: remplacer par appel HTTP …. Le branchement sur la vraie API ne nécessite de modifier que ces fichiers — aucun composant n'est à toucher.
